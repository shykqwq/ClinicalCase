# 乳腺外科临床病例库与科研随访 App 需求规格说明书（P0 版）

版本：v0.1  
日期：2026-06-14  
适用范围：Android App + 后端 API + 云数据库 + 对象存储  
目标阶段：P0 首版开发

## 1. 项目背景

乳腺外科临床工作中，患者资料分散在病历系统、检查报告、影像报告、病理报告、随访记录和医生个人笔记中。对于科室内部病例积累、科研筛选、术后随访和治疗结局追踪，传统表格或个人文档难以满足多人协作、结构化检索和长期维护的需求。

本项目拟建设一套面向乳腺外科科室内部使用的临床病例库与科研随访系统。首版以 Android App 为主要入口，配套后端 API、云数据库和对象存储，实现病例录入、附件归档、科研筛选、随访管理、多人协作和基础权限控制。

## 2. 产品定位

本 App 不是医院 HIS/EMR 系统替代品，也不在 P0 阶段对接医院信息系统。它定位为科室内部使用的乳腺外科专病病例库与科研随访工具。

核心目标：

- 建立结构化乳腺外科病例库。
- 支持科室多人协作维护病例资料。
- 支持按乳腺专病字段进行科研筛选。
- 支持术后、治疗后和长期随访管理。
- 支持患者相关报告、图片和 PDF 资料归档。
- 在存储真实患者身份信息的前提下提供基础安全与权限控制。

## 3. 目标用户

### 3.1 科室管理员

通常为项目负责人或科室指定管理员，负责成员管理、权限分配、病例维护监督和数据导出。

### 3.2 临床医生

负责录入、查看、编辑患者病例资料，维护诊疗过程、病理信息、治疗方案和随访记录。

### 3.3 研究助手

负责补充资料、上传附件、维护随访、标记科研标签，但不具备删除病例和管理成员的权限。

### 3.4 只读成员

用于教学、病例讨论或临时查看，只允许查看授权范围内病例，默认查看脱敏信息。

## 4. P0 业务目标

P0 首版需要完成从“患者建档”到“病例归档、筛选和随访”的最小闭环：

1. 科室成员可以登录系统。
2. 管理员可以管理科室成员和角色。
3. 医生可以创建患者档案。
4. 医生可以录入乳腺外科专病病例信息。
5. 用户可以上传和查看病例附件。
6. 用户可以按关键字段搜索和筛选病例。
7. 用户可以创建、查看和更新随访计划与随访记录。
8. 系统可以云端保存数据，并支持 Android 端查看最新数据。
9. 系统具备基础权限、脱敏显示、操作日志和数据备份能力。

## 6. 系统架构

### 6.1 总体架构

系统由四部分组成：

- Android App：医生和科室成员的主要操作入口。
- 后端 API：提供认证、权限、病例、附件、随访和审计接口。
- 云数据库：保存结构化业务数据。
- 对象存储：保存图片、PDF、报告附件等非结构化文件。

### 6.2 推荐技术栈

Android 端：

- Kotlin
- Jetpack Compose
- MVVM
- Retrofit
- Room
- DataStore
- WorkManager

后端：

- Spring Boot / NestJS / Django 三选一
- RESTful API
- JWT 或 Session Token 认证
- RBAC 权限控制

数据库：

- PostgreSQL 优先
- MySQL 可作为备选

对象存储：

- 阿里云 OSS / 腾讯云 COS / AWS S3 兼容存储均可

## 8. 功能需求

### 8.1 登录与认证

功能说明：

- 用户通过账号密码登录。
- 登录成功后获取访问令牌。
- Android 端保存登录状态。
- 令牌过期后需要重新登录或刷新令牌。

字段：

- 账号
- 密码
- 用户姓名
- 所属科室
- 角色
- 账号状态

验收标准：

- 有效账号可以登录。
- 无效账号或错误密码不能登录。
- 停用账号不能登录。
- 登录后可以获取当前用户信息。
- 未登录访问业务接口时返回未授权。

### 8.2 科室成员管理

功能说明：

- 管理员可以新增成员。
- 管理员可以编辑成员姓名、角色和状态。
- 管理员可以停用成员账号。
- P0 不要求开放自助注册。

成员状态：

- 启用
- 停用

角色：

- 科室管理员
- 临床医生
- 研究助手
- 只读成员

验收标准：

- 管理员可以创建成员账号。
- 非管理员不能创建成员。
- 停用成员后，该成员不能再登录。
- 成员角色变更后，新权限立即生效或在下次登录后生效。

### 8.3 患者档案管理

功能说明：

- 创建患者基础档案。
- 查看患者详情。
- 编辑患者基础信息。
- 支持按姓名、手机号、住院号、门诊号搜索。
- 支持脱敏显示。

P0 字段：

- 患者 ID
- 姓名
- 性别
- 出生日期
- 年龄
- 手机号
- 身份证号，可选
- 门诊号
- 住院号
- 初诊日期
- 主治医生
- 建档医生
- 备注
- 创建时间
- 更新时间

脱敏规则：

- 姓名：显示姓氏 + “**”，例如“张**”。
- 手机号：显示前三位和后四位。
- 身份证号：仅显示前六位和后四位。

验收标准：

- 医生可以创建患者。
- 患者姓名、性别、初诊日期为必填。
- 同一科室内住院号或门诊号重复时应提示用户确认。
- 只读成员默认看不到完整手机号和身份证号。

### 8.4 乳腺外科专病病例管理

功能说明：

- 每个患者可关联一个或多个病例。
- 病例包含乳腺外科专病结构化字段。
- 支持保存草稿和正式归档状态。

P0 字段：

- 病例 ID
- 患者 ID
- 病例标题
- 就诊类型：门诊、住院、术后复查
- 侧别：左乳、右乳、双侧、不详
- 疾病类型：乳腺癌、乳腺良性肿瘤、乳腺炎、其他
- 初步诊断
- 确定诊断
- 病理类型
- ER 状态：阳性、阴性、不详
- PR 状态：阳性、阴性、不详
- HER2 状态：0、1+、2+、3+、FISH 阳性、FISH 阴性、不详
- Ki-67 数值
- 分子分型：Luminal A、Luminal B、HER2 阳性、三阴性、不详
- 临床 T 分期
- 临床 N 分期
- 临床 M 分期
- 病理 T 分期
- 病理 N 分期
- 病理 M 分期
- 临床分期
- 病理分期
- 是否新辅助治疗
- 新辅助治疗方案
- 手术日期
- 手术方式
- 腋窝处理方式
- 化疗方案
- 是否放疗
- 是否靶向治疗
- 靶向治疗方案
- 是否内分泌治疗
- 内分泌治疗方案
- 复发状态：无复发、局部复发、远处转移、不详
- 当前状态：治疗中、随访中、复发、死亡、失访
- 病例摘要
- 病例状态：草稿、已归档
- 创建人
- 更新人
- 创建时间
- 更新时间

验收标准：

- 一个患者可以创建多个病例。
- 必填字段缺失时不能归档。
- 草稿可以保存不完整信息。
- 已归档病例仍可编辑，但需要记录操作日志。
- 病例详情可以展示患者信息、诊断信息、治疗信息、附件和随访摘要。

### 8.5 附件管理

功能说明：

- 用户可以为患者或病例上传附件。
- 附件保存到对象存储。
- 数据库保存附件元数据。
- Android 端支持图片和 PDF 预览。

附件分类：

- 病理报告
- 影像报告
- 检验报告
- 手术资料
- 出院小结
- 门诊病历
- 随访资料
- 其他

支持格式：

- JPG
- PNG
- PDF

P0 限制：

- 单个文件最大 20 MB。
- 单次最多上传 9 个文件。
- 视频暂不支持。

字段：

- 附件 ID
- 患者 ID
- 病例 ID，可为空
- 附件分类
- 文件名
- 文件类型
- 文件大小
- 对象存储 Key
- 上传人
- 上传时间

验收标准：

- 用户可以上传图片和 PDF。
- 上传成功后可在病例详情中查看。
- 附件删除后业务列表不再显示。
- 对象存储访问应通过后端授权 URL 或代理接口完成，不能直接公开永久访问地址。

功能说明：

P0 搜索条件：

- 姓名

- 门诊号

P0 筛选条件：

- 初诊日期范围
- 疾病类型
- 侧别
- 病理类型
- ER 状态
- PR 状态
- HER2 状态

- 临床分期
- 病理分期
- 手术方式
- 是否新辅助治疗
- 是否化疗
- 是否放疗
- 是否靶向治疗
- 是否内分泌治疗
- 复发状态
- 当前状态
- 科研标签

验收标准：

- 输入姓名或住院号可以快速找到患者。
- 多个筛选条件可以组合使用。

## 10. 后端 API 模块

P0 API 模块：

- Auth API：登录、刷新令牌、退出登录、当前用户信息。
- User API：成员列表、新增成员、编辑成员、停用成员。
- Patient API：患者新增、查询、详情、编辑、删除。
- Case API：病例新增、查询、详情、编辑、归档、删除。
- Attachment API：上传授权、附件保存、附件列表、附件预览、附件删除。
- Tag API：标签新增、列表、编辑、删除、病例标签绑定。
- Follow-up API：随访计划、随访记录、待随访列表、逾期随访列表。
- Audit API：操作日志列表、操作日志详情。

## 12. 数据安全与合规要求

由于系统存储真实患者身份信息，P0 即需要满足基础安全要求：

- 全部接口必须使用 HTTPS。
- 密码必须使用不可逆哈希保存。
- 访问接口必须经过身份认证。
- 后端必须校验用户角色和数据权限。
- 患者姓名、手机号、身份证号等敏感字段应支持脱敏显示。
- 附件访问不能使用永久公开链接。
- 数据库需要定期备份。
- 关键操作需要记录审计日志。
- Android 端本地缓存不应明文保存密码。
- Android 端本地缓存中的敏感数据应尽量减少。
- 退出登录后应清理本地令牌。

备注：本系统为科室内部工具，仍应遵守医院信息安全制度、患者隐私保护要求和个人信息保护相关法规。上线前应根据实际部署环境补充安全评估。

## 13. 非功能需求

### 13.1 性能

- 患者列表首屏加载时间不超过 2 秒。
- 常规搜索响应时间不超过 3 秒。
- 单张图片上传在正常网络下应有明确进度反馈。

### 13.2 可用性

- 表单字段应按临床工作流程分组。
- 必填字段应清晰标识。
- 保存失败需要明确提示原因。
- 网络异常时应提示用户重试。

### 13.3 可靠性

- 后端接口应避免重复提交造成重复病例。
- 附件上传失败时不应产生无效业务记录。
- 随访计划生成失败时应提示用户。

### 13.4 可维护性

- 后端 API 应提供清晰版本号，例如 `/api/v1`。
- 枚举字段应统一维护。
- 操作日志和业务表应分离。
- 附件元数据和对象存储文件应可追踪。

## 14. P0 验收清单

P0 完成标准：

- 管理员可以登录并创建科室成员。
- 临床医生可以登录并创建患者。
- 临床医生可以创建乳腺外科专病病例。
- 病例可以保存草稿并归档。
- 用户可以上传图片和 PDF 附件。
- 用户可以查看患者详情、病例详情、附件和随访记录。
- 用户可以按姓名、住院号和乳腺专病字段筛选病例。
- 用户可以创建随访计划并完成随访记录。
- 首页可以展示待随访、逾期随访和最近病例。
- 只读成员默认看到脱敏信息。
- 关键操作可以写入审计日志。
- Android App 通过后端 API 获取和提交数据。
- 数据保存在云数据库中。
- 附件保存在对象存储中。

## 15. P1/P2 后续规划

P1：

- Excel 数据导出。
- 脱敏病例导出。
- 病例 PDF 生成。
- 高级统计图表。
- 批量导入历史病例。
- Web 管理后台。

P2：

- 科研队列管理。
- 患者端随访问卷。

## 16. 待确认问题

进入详细设计前，需要进一步确认：

1. 后端技术栈最终选择 Spring Boot、NestJS 还是 Django。
2. 云服务选择阿里云、腾讯云、华为云还是其他平台。
3. 是否需要 Android 离线编辑能力，还是只需要在线访问。
4. 患者身份证号是否为 P0 必填，当前建议为可选。
5. 是否允许医生查看全科室所有病例，还是只查看自己负责和授权病例。
6. 附件是否需要服务端病毒扫描。
7. 是否需要水印、防截图或导出审批。
